model Run — one record per execution (full workflow, partial, or single node)
Field Type Req Notes
id PK String @default(cuid()) — collision-resistant unique ID
workflowId FK String References Workflow.id. Auto-upsert the workflow record on every run using workflowName + userId.
userId FK String Clerk user ID from auth() in the API route
status RunStatus Enum: running success failed partial
scope RunScope Enum: full partial single
triggerRunId String? The Trigger.dev run ID — stored so the subscription can be matched back to the DB record
startedAt DateTime @default(now()) — set automatically on creation
completedAt DateTime? Written by PATCH /api/runs/[id] when the Trigger.dev run ends
durationMs Int? Computed: completedAt - startedAt in milliseconds
model NodeRun — one record per node per execution
Field Type Req Notes
id PK String @default(cuid())
runId FK String References Run.id
nodeId String React Flow node ID (e.g. uploadImageNode-1714200000000)
nodeType String e.g. uploadImageNode, runLLMNode, cropImageNode
status NodeStatus Enum: pending running success failed
input Json? Snapshot of the node's data at the time the run started. Captured client-side from React Flow state.
output Json? Written when the node succeeds. Sourced from run.metadata.nodeOutputs[nodeId] (set by the orchestrator task).
error String? Error message string if the node failed. From run.metadata.nodeErrors[nodeId].
startedAt DateTime @default(now()) — set when the NodeRun row is created
completedAt DateTime? Set by PATCH route. From run.metadata.nodeDurations[nodeId].
durationMs Int? Per-node execution time in ms. Set during the PATCH update.

03 — Orchestrator Metadata

What the orchestrator must write to metadata

The orchestrator already writes nodeStatuses. It needs three additional maps so the API route can populate NodeRun records completely on completion. 

// Inside src/trigger/orchestrator.ts — metadata shape
await metadata.set("nodeStatuses",  { "nodeId-1": "success", "nodeId-2": "failed" });
await metadata.set("nodeOutputs",  { "nodeId-1": { image_url: "..." } });
await metadata.set("nodeErrors",   { "nodeId-2": "Connection timeout" });
await metadata.set("nodeDurations",{ "nodeId-1": 1240, "nodeId-2": 320 }); // ms

TypeScript shape

interface OrchestratorMetadata {
  nodeStatuses:   Record<string, "pending" | "running" | "success" | "failed">;
  nodeOutputs:    Record<string, unknown>;
  nodeErrors:     Record<string, string>;
  nodeDurations:  Record<string, number>;
}
04 — API Routes

Three new endpoints

All routes live under src/app/api/runs/ and use auth() from Clerk for authentication, identical to the existing /api/nodes/run route.

POST /api/runs Create run record

Called immediately after Trigger.dev returns { runId, publicToken }. Creates the Run row and one NodeRun row per node (all in pending status) in a single transaction. Also upserts the Workflow record.

Request body

{
  "triggerRunId": "run_abc123",
  "workflowName": "My Workflow",
  "scope":        "full",
  "nodes": [
    { "id": "uploadImageNode-1", "type": "uploadImageNode", "data": { ... } },
    { "id": "runLLMNode-2",      "type": "runLLMNode",      "data": { ... } }
  ],
  "edges": [ ... ]
}

Response

{
  "dbRunId": "clxyz1234"
}

The dbRunId is stored in React state alongside the Trigger.dev run so the PATCH call knows which DB record to update.

Prisma operations (in a transaction)

// 1. Upsert User (Clerk might be ahead of DB)
prisma.user.upsert({ where: { id: userId }, ... });

// 2. Upsert Workflow by name + userId
prisma.workflow.upsert({
  where: { userId_name: { userId, name: workflowName } },
  update: { nodes, edges, updatedAt: now },
  create: { id: cuid(), name: workflowName, userId, nodes, edges },
});

// 3. Create Run
prisma.run.create({
  data: { workflowId, userId, status: "running", scope, triggerRunId }
});

// 4. Create one NodeRun per node (createMany)
prisma.nodeRun.createMany({
  data: nodes.map(n => ({
    runId, nodeId: n.id, nodeType: n.type,
    status: "pending", input: n.data,
  }))
});
PATCH /api/runs/[id] Complete run record

Called from the useEffect subscription in DashboardShell when run.isCompleted or run.isFailed. Writes final status, duration, and all node-level results.

Request body

{
  "status":      "success",
  "completedAt": "2026-04-27T10:30:00Z",
  "nodeResults": {
    "uploadImageNode-1": {
      "status":     "success",
      "output":     { "image_url": "..." },
      "durationMs": 1240
    },
    "runLLMNode-2": {
      "status":     "failed",
      "error":      "Rate limit exceeded",
      "durationMs": 320
    }
  }
}

Prisma operations

// Update the Run row
prisma.run.update({
  where: { id: dbRunId },
  data: { status, completedAt, durationMs }
});

// Update each NodeRun row
for (const [nodeId, result] of
     Object.entries(nodeResults)) {
  prisma.nodeRun.updateMany({
    where: { runId: dbRunId, nodeId },
    data: {
      status:      result.status,
      output:      result.output,
      error:       result.error,
      durationMs:  result.durationMs,
      completedAt: now,
    }
  });
}
GET /api/runs Fetch run history

Called on sidebar mount and after each run completes. Returns the 20 most recent runs for the current user, with nested node runs.

Prisma query

prisma.run.findMany({
  where:   { userId },
  orderBy: { startedAt: "desc" },
  take:    20,
  include: {
    nodeRuns: {
      orderBy: { startedAt: "asc" }
    },
    workflow: {
      select: { name: true }
    }
  }
});

Response shape

[
  {
    "id":          "clxyz1234",
    "status":      "success",
    "scope":       "full",
    "startedAt":   "2026-04-27T10:29:00Z",
    "durationMs":  3480,
    "workflow":    { "name": "My Workflow" },
    "nodeRuns": [
      {
        "nodeId":    "uploadImageNode-1",
        "nodeType":  "uploadImageNode",
        "status":    "success",
        "durationMs":1240,
        "output":    { ... },
        "input":     { ... }
      }

]
05 — Integration Hooks

Where to wire calls into existing code

  • 1

    DashboardShell — handleRunWorkflow()

    After runWorkflowFnRef.current() resolves with { runId, publicToken }, immediately fire POST /api/runs with the current node/edge list, workflow name, and scope "full". Store the returned dbRunId in state alongside workflowRun.

  • 2

    DashboardShell — useEffect (orchestrator subscription)

    On run.isCompleted or run.isFailed, read run.metadata (nodeOutputs, nodeErrors, nodeDurations) and call PATCH /api/runs/[dbRunId] with the assembled payload. Then call refreshHistory() to re-fetch the sidebar list.

  • 3

    Individual node handlers (single-node runs)

    Each node's handleUpload / handleRun already calls /api/nodes/run. After getting back { runId, publicToken }, also call POST /api/runs with scope: "single" and a one-item nodes array. The existing RunStatus component already subscribes to completion — hook the PATCH call there too.

  • 4

    RightBar — history tab

    On mount, call GET /api/runs and store results in local state. Expose a refreshHistory callback passed down from DashboardShell so the sidebar re-fetches whenever a run finishes. No polling needed — only refresh on known state transitions.
